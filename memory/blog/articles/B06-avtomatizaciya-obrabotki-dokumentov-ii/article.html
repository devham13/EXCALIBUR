Счета от поставщиков снова лежат в почте, а бухгалтер вручную переносит ИНН и суммы в CRM? Автоматизация обработки документов с ИИ – это не покупка СЭД, а связка: PDF или скан → распознавание → JSON с реквизитами → проверка → ручное approve → запись в amoCRM или Bitrix24. За 1–2 недели пилота вы соберёте такой конвейер на n8n или Make без штата разработчиков и замерите KPI на 30–50 реальных счетах.

TL;DR / Быстрый инсайт: Рабочая схема: intake (папка или email) → OCR/parse → LLM в strict JSON → валидация ИНН и сумм → human-in-the-loop → CRM API. OCR даёт текст, LLM структурирует поля – на счетах нужны оба слоя. Автономные ИИ-системы закрывают менее 2,5% сложных задач без человека, поэтому approve перед проводкой обязателен. Успешный пилот: ≥90% верных обязательных полей на выборке 30–50 документов.

IDP (Intelligent Document Processing – "умная обработка документов") в 2026 году – это не один OCR-сервис, а цепочка с контролем ошибок. Запрос "автоматизация обработки документов" (около 599 показов в месяц по данным Вордстата, июнь 2026) чаще означает путь до учётки, а не внедрение ЭДО. Ниже – один тип документа (входящие счета), один конвейер, измеримый результат.

Webhook – входящий "звонок" на URL вашего сценария: файл пришёл, workflow стартует. API (Application Programming Interface) – стандартная "розетка" для записи данных в CRM без копипаста.

1. Составьте карту workflow от intake до CRM

Схема конвейера:
Google Drive / email IMAP → Extract from PDF или OCR → LLM(JSON) → validate(ИНН, НДС, hash) → Switch → [approve в Sheets/Telegram] → amoCRM / Bitrix24 API → алерт при ошибке

Не пытайтесь автоматизировать все типы документов сразу. На пилоте – только счета от поставщиков: один источник (папка или inbox), одна CRM или Google Sheets, 30–50 тестовых PDF и сканов.

Делайте: фиксируйте каждый этап в журнале (raw OCR, JSON, статус, кто approve). Не делайте: пять каналов intake без логов – вы не поймёте, где ломается цепочка.

2. Выберите стек: n8n vs Make и OCR vs LLM

Критерий n8n Make.com
Старт без сервера Cloud trial или self-host Регистрация за минуты, 2500+ интеграций
PDF → текст Узел Extract from PDF (встроенный) Модули PDF + HTTP к OCR API
Валидация ИНН Code-узел с checksum Custom function или HTTP к своему скрипту
152-ФЗ / ПДн Self-host + GigaChat/YandexGPT Облако Make; чувствительные PDF – on-prem LLM
CRM HTTP Request к amoCRM v4 / Bitrix24 Нативные модули amoCRM, Bitrix24

Для цифровых PDF (текст внутри файла) – n8n Extract from PDF. Для фото сканов – OCR API или multimodal LLM (модель, которая "видит" картинку). По данным VibeLab, точность OCR на типовых российских документах при чистом скане – 94–97%.

Делайте: Make, если команда no-code и нужен быстрый старт с amoCRM. Не делайте: отправлять PDF с реквизитами контрагентов в зарубежные LLM без правовой оценки – для ПДн предпочтите GigaChat, YandexGPT или on-prem.

Итоговый вердикт: n8n – гибкость, self-host и Code-узлы для валидации. Make – скорость CRM-интеграций без Docker. Подробнее про оркестраторы – в гайде автоматизация n8n с ИИ-агентами.

3. Соберите конвейер пошагово: от триггера до CRM

  1. Шаг 1: Аудит. Выберите один тип – счета. Соберите 30–50 файлов (PDF и сканы), определите целевую CRM или таблицу и поля для экспорта.
  2. Шаг 2: Intake. Настройте триггер: Google Drive (новый файл в папке), IMAP Gmail или webhook от почтового сервиса. Сохраняйте binary PDF, не только ссылку.
  3. Шаг 3: Parse. Цифровой PDF – Extract from PDF в n8n. Скан – OCR/Vision API или multimodal LLM с temperature 0.
  4. Шаг 4: Extract. LLM с JSON-schema: поставщик (ИНН, КПП), номер и дата счёта, net/VAT/gross, валюта. Structured output, без свободного текста "перескажи счёт".
  5. Шаг 5: Validate. Code-узел: checksum ИНН, арифметика сумм (net + НДС = gross), hash(ИНН+номер+дата) против дубликатов. Порог confidence – например 85%.
  6. Шаг 6: Route. Switch: если validation OK и confidence ≥ порога – auto-export; иначе – очередь review (Google Sheets или Telegram с кнопкой approve).
  7. Шаг 7: Export. HTTP Request: amoCRM /api/v4/leads или Bitrix24 crm.deal.add. Либо строка в Sheets как промежуточный буфер.
  8. Шаг 8: Monitor. При fail validation – Telegram/Slack. Журнал: raw OCR + JSON + status + operator id для бухгалтерии.

Типовой срок сборки MVP intake→CRM на одном типе документа – 1–2 недели (ориентир, не гарантия). Главный бухгалтер по оценке VibeLab тратит до 60% времени на сверку и классификацию – автоматизация бьёт именно в этот слой.

4. Настройте JSON-schema и промпт для LLM

LLM без схемы вывода "придумывает" поля. В 2026 году стандарт IDP – strict JSON schema + temperature 0 + проверка на стороне кода.

{
  "supplier_inn": "string",
  "supplier_kpp": "string",
  "invoice_number": "string",
  "invoice_date": "YYYY-MM-DD",
  "amount_net": "number",
  "amount_vat": "number",
  "amount_gross": "number",
  "currency": "RUB",
  "confidence": "number"
}

Промпт: "Извлеки поля счёта строго по schema. Если поле не видно – null, не выдумывай. Верни только JSON." Human-in-the-loop (HITL) – обязательная пауза: человек подтверждает спорные счета перед проводкой в учёт.

Делайте: на пилоте экспортируйте только обязательные поля, строки позиций – позже. Не делайте: auto-export при confidence ниже 80%.

5. Проверьте валидацию и подключите human-in-the-loop

ИНН проверяется контрольной суммой в Code-узле – при fail документ уходит в review, не в CRM. НДС: gross должна сходиться с net + VAT с допуском на округление. Hash по ИНН+номер+дата ловит повторную загрузку того же счёта.

Очередь approve: строка в Google Sheets (статус "ожидает") или сообщение в Telegram с кнопками "принять / отклонить". Только после approve – HTTP в CRM. Менее 2,5% сложных неструктурированных задач закрываются полностью без человека – для бухгалтерии это не риск, а норма.

Делайте: логируйте operator id при каждом approve. Не делайте: слепую запись в CRM сразу после OCR без второго слоя проверки.

6. Пройдите чеклист пилота на 30–50 счетах

  • Один тип документа и один канал intake
  • 30–50 реальных PDF и сканов в тестовой выборке
  • JSON-schema с обязательными полями зафиксирована
  • Checksum ИНН и арифметика сумм в Code-узле
  • Anti-duplicate hash включён
  • Switch по confidence настроен
  • Очередь HITL (Sheets или Telegram) работает
  • Экспорт в amoCRM / Bitrix24 / Sheets протестирован
  • Алерт при fail validation в мессенджер
  • Журнал raw OCR + JSON + status ведётся
  • 152-ФЗ: чувствительные PDF не уходят в зарубежные LLM без оценки
  • KPI: % верных полей, время intake→CRM, число эскалаций
  • Критерий успеха: ≥90% верных обязательных полей

Если Make ближе вашей команде – первые три занятия по сценариям intake и HTTP доступны в курсе Make.com и вайбкодинга на kv-ai.ru.

Частые вопросы

Чем OCR отличается от LLM extraction при обработке счетов?

OCR превращает картинку или PDF в текст. LLM структурирует текст в поля JSON (ИНН, суммы, даты). На счетах используйте оба слоя или multimodal модель с vision. Настройте Extract from PDF для цифровых файлов и OCR для сканов.

Можно ли обойтись без human-in-the-loop?

Нет для бухгалтерии и CRM: автономные системы закрывают менее 2,5% сложных задач без человека. Добавьте очередь approve в Sheets или Telegram перед записью в учёт.

n8n или Make для извлечения данных из PDF?

Make быстрее старт с облачными CRM и 2500+ интеграций. n8n гибче для self-host, Code-узлов validation и шаблона #2764 (PDF→LLM). Выберите Make для no-code команды, n8n – если нужен свой сервер.

Какие поля счёта класть в JSON-schema на пилоте?

Минимум: ИНН и КПП поставщика, номер и дата счёта, net/VAT/gross, валюта, confidence. Строки позиций добавьте после достижения ≥90% на обязательных полях.

Как автоматически проверить ИНН?

Реализуйте контрольную сумму в Code-узле n8n или custom function в Make. При fail направляйте документ в review, не в CRM API.

Куда отправлять PDF по 152-ФЗ?

Персональные данные контрагентов – в GigaChat, YandexGPT или on-prem LLM. Не загружайте реквизиты в зарубежные облачные модели без правовой оценки и политики компании.

Когда пилот считается успешным?

≥90% верных обязательных полей на 30–50 счетах, измеримый SLA intake→CRM, все спорные документы прошли approve. Только после этого добавляйте новые типы документов или каналы intake.
