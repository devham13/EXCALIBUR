Agent в Cursor отвечает по-разному в каждом чате, пока вы не зафиксировали cursor rules – постоянные инструкции, которые редактор подмешивает в начало контекста модели. Файл .cursorrules в корне устарел; в 2026 рабочий формат – папка .cursor/rules/ с файлами .mdc и YAML-frontmatter. За 30–45 минут вы создадите первые правила, выберете режим alwaysApply или globs, проверите активацию через context gauge и закоммитите набор в Git по чеклисту.

TL;DR / Быстрый инсайт: Cursor Rules – текстовые инструкции в .cursor/rules/*.mdc с YAML-шапкой (frontmatter). Поля alwaysApply, globs и description задают, когда правило попадает в Agent. Один глобальный alwaysApply + scoped-правила по glob экономят контекст. Проверка – hover у поля ввода Agent (context gauge) или вызов @имя-правила. Rules не заменяют MCP: правила задают поведение, MCP дает доступ к внешним сервисам.

Cursor Rules – "устав проекта" для ИИ: язык, стек, запреты, стиль кода или SEO. По документации Cursor текст правила попадает в начало контекста; plain .md в .cursor/rules/ игнорируется – для markdown без YAML есть AGENTS.md. Настройка cursor rules удобна командам автоматизации: общие правила для n8n и контента в Git. Связка с MCP в Cursor: rules – как думать, MCP – к чему подключаться.

1. Проверьте legacy и мигрируйте с .cursorrules

Если в корне репозитория лежит файл .cursorrules – формат помечен как deprecated. Новые правила создавайте только как .mdc в .cursor/rules/. User Rules живут в настройках Cursor на вашем ПК и не попадают в Git; Project Rules – как раз то, что коммитится в репозиторий.

Делайте: скопируйте содержимое .cursorrules в новый файл 00-general.mdc с alwaysApply: true, затем удалите legacy-файл, чтобы Agent не путался между двумя источниками. Не делайте: хранить и .cursorrules, и дублирующий alwaysApply – двойной расход контекста на каждый запрос.

2. Создайте первое правило .mdc с frontmatter

Project rules – файлы с расширением .mdc в каталоге .cursor/rules/ у корня репозитория. Создать папку можно вручную, через Cursor Settings → Rules, Commands → + Add Rule или командой /create-rule в чате Agent – но сгенерированный frontmatter всегда сверяйте с таблицей режимов ниже.

Workflow настройки:
Проверка .cursorrules → mkdir .cursor/rules/ → 00-general.mdc (alwaysApply) → glob-правило → intelligent-правило → тест в Agent → modular rules → git commit

Пример базового правила для всего проекта (адаптируйте стек и язык под себя):

---
description: Общие правила проекта
alwaysApply: true
---

# Общие правила

- Отвечай на русском, код и пути – на английском.
- Стек: Make/n8n, Cursor Agent, Markdown для контента.
- Не добавляй секреты, API-ключи и пароли в ответы и коммиты.
- Перед правкой файлов – кратко опиши план из 2–3 шагов.
- Держи ответы компактными; длинные списки – таблицей или списком.

Best practice Cursor: одно правило короче 500 строк; при росте – дробите на несколько .mdc. Делайте: начинайте с 00-general.mdc на 150–200 слов. Не делайте: класть весь "устав" в один alwaysApply на 2000 строк – Agent получит лишний шум в каждом чате.

  1. Шаг 1: Создайте каталог .cursor/rules/ в корне git-репозитория.
  2. Шаг 2: Добавьте 00-general.mdc с тремя чертами --- до и после YAML-frontmatter.
  3. Шаг 3: Установите alwaysApply: true только для этого файла.
  4. Шаг 4: Сохраните файл и откройте Agent в том же проекте.
  5. Шаг 5: Наведите курсор на context gauge у поля ввода – должно отображаться активное правило.

3. Настройте четыре режима активации правил

Три поля frontmatter – description, globs, alwaysApply – определяют, когда cursor rules mdc попадает в контекст. При alwaysApply: true поля globs и description игнорируются. При alwaysApply: false возможны три сценария: по glob-маске файлов, по описанию (Agent решает сам – Apply Intelligently) или только вручную через @rule-name в чате.

Режим Frontmatter Когда срабатывает
Always Apply alwaysApply: true В каждый чат Agent в этом проекте
Apply to Files alwaysApply: false + globs Когда в контексте есть файл по маске (*.ts, workflows/**/*.json)
Apply Intelligently alwaysApply: false + description, без globs Agent оценивает задачу по тексту description
Manual alwaysApply: false, без globs и description Только если вы написали @имя-правила в чате

Несколько glob-паттернов – через запятую в docs или YAML-список на форуме; движок – minimatch. Scoped-правило для n8n: globs: workflows/**/*.json, alwaysApply: false, в теле – конвенции имен нод. Делайте: alwaysApply: true только для языка и запретов. Не делайте: включать alwaysApply на все rules – Tab и Inline Edit (Cmd/Ctrl+K) rules не используют.

4. Соберите набор rules для автоматизации контента

Минимальный стек для контент-завода и no-code автоматизации – 3–4 файла, а не "топ-100 rules" из чужих репозиториев:

Файл Режим Назначение
00-general.mdc Always Apply Язык, стек, запреты
frontend-components.mdc globs: src/**/*.tsx Стиль UI-компонентов
n8n-workflows.mdc globs: workflows/**/*.json Конвенции сценариев
content-seo.mdc Intelligent (description про SEO, H1–H2, internal links) Статьи и meta
release-checklist.mdc Manual (@release-checklist) Чеклист перед релизом

Приоритет при конфликте: Team Rules → Project Rules → User Rules. Rules – как Agent отвечает; MCP – какие внешние tools доступны (гайд MCP); Skills – готовые сценарии внутри Cursor. Делайте: именуйте файлы по домену. Не делайте: копировать чужие rules без правки glob-путей.

5. Пройдите чек-лист из 15 пунктов перед коммитом в Git

  • Каталог называется .cursor/rules/, не .cursor/rule/
  • Расширение файлов – .mdc, не .md (plain .md в rules игнорируется)
  • YAML между --- валиден: отступы пробелами, без табов
  • alwaysApply: true – максимум 1–2 файла на проект
  • Каждый glob проверен на реальном файле через context gauge
  • Intelligent-правила имеют четкий description на 1–2 предложения
  • Manual-правила документированы в README команды (@имя)
  • Нет API-ключей, токенов и паролей в теле rules
  • Legacy .cursorrules удален после миграции
  • Имена файлов осмысленные, с префиксом 00- для general
  • Каждое правило короче 500 строк
  • .cursor/rules/ не в .gitignore (если rules – командный стандарт)
  • Тестовый prompt в Agent подтвердил срабатывание glob/intelligent
  • Team Rules на dashboard не дублируют project rules без нужды
  • Коммит-сообщение описывает, что изменилось в поведении Agent

Делайте: коммитьте rules вместе с кодом, который они описывают. Не делайте: пушить rules с placeholder-текстом "TODO" – Agent будет следовать мусору буквально.

6. Исправьте типичные ошибки, если правило не срабатывает

  1. Шаг 1: Проверьте путь .cursor/rules/ и расширение .mdc.
  2. Шаг 2: Убедитесь, что frontmatter закрыт второй строкой --- без лишних символов.
  3. Шаг 3: Для glob-режима добавьте в контекст (@) файл, который попадает под маску.
  4. Шаг 4: Для Intelligent-режима уточните prompt словами из description.
  5. Шаг 5: Для Manual – @имя-файла; сверьте context gauge после правки frontmatter.

Делайте: Reload Window после смены YAML. Не делайте: ждать работы rules в Tab или Inline Edit – только Agent (Chat).

7. Что делать дальше

  1. Закоммитьте минимальный набор: general + один glob + один intelligent или manual.
  2. Подключите MCP под ваши сервисы – см. инструкцию по MCP.
  3. Добавьте content-seo.mdc, если ведете контент-завод или SEO-блог.
  4. Раз в месяц ревизируйте alwaysApply – не раздувается ли общий контекст.
  5. Для сценариев Make/n8n без кода – обучение по Make.com с разбором AI-агентов.

Частые вопросы

Чем Cursor Rules отличаются от MCP?

Rules – постоянные текстовые инструкции в контексте Agent (язык, стиль, запреты). MCP (Model Context Protocol) – "розетки" для внешних tools: браузер, GitHub, CMS. Настройте оба слоя: rules в .cursor/rules/, MCP в mcp.json – по гайду B03.

Как написать cursor rules для проекта?

Создайте .cursor/rules/, добавьте файлы .mdc с YAML-frontmatter (alwaysApply, globs или description), проверьте в Agent через context gauge и закоммитьте в Git. Стартуйте с одного alwaysApply и одного glob-правила.

Когда ставить alwaysApply: true?

Только для глобального минимума: язык ответов, стек, критичные запреты (секреты, стиль коммитов). Доменные правила (SEO, n8n, UI) держите на globs или Intelligent – так вы не сжигаете контекст на каждый вопрос.

Почему правило не применяется?

Частые причины: неверный glob, файл .md вместо .mdc, битый YAML между ---, Intelligent без подходящего prompt. Откройте context gauge, добавьте нужный файл в @-контекст или вызовите @rule вручную.

Нужно ли удалять .cursorrules?

Да, после переноса текста в .mdc с alwaysApply: true. Формат .cursorrules deprecated; два источника правил создают путаницу и лишний расход токенов.

Rules или AGENTS.md – что выбрать?

.mdc – scoped rules с frontmatter; AGENTS.md – простой markdown в корне или подпапках. Выберите один формат для одного текста.

Rules vs Skills в Cursor – в чем разница?

Rules (.mdc) – рамки проекта; Skills – пошаговый сценарий агента. Rules постоянны; Skills – под конкретную задачу.
