Мультиагентная система в n8n разделяет роли: orchestrator (дирижёр) решает, кому передать задачу, specialists выполняют узкие действия. С версии 1.82.0 все AI Agent работают как Tools Agent и требуют минимум один tool sub-node; второй полноценный agent подключается как AI Agent Tool с собственной Chat Model, System Message и tools.

Менее 2,5% сложных неструктурированных задач автономные системы закрывают без human-in-the-loop - планируйте эскалацию заранее.

Проверьте, когда одного AI Agent мало

Multi-agent оправдан в трёх типичных сценариях:

  • Смешанные системы: база знаний + CRM + эскалация менеджеру в одном диалоге.
  • Разные модели: gpt-4o-mini для routing, Claude для финального текста.
  • Reflexion: worker пишет черновик, critic проверяет - отдельные prompt и max iterations.

Anti-pattern: MAS вместо одного agent с RAG и 2-3 tools. Coordination overhead: 3 агента - 3 связи, 10 агентов - 45 (n8n.io). Официальный tutorial n8n строит hierarchical MAS: main agent + email sub-agent + RAG sub-agent - тот же принцип для бизнес-кейсов.

Делайте: выпишите 5 реальных диалогов и отметьте, сколько систем задействовано. Не делайте: sub-agents "на будущее" без задачи для каждого.

Выберите паттерн оркестрации под ваш кейс

Паттерн Когда Ноды n8n
Supervisor Непредсказуемый запрос, динамическое делегирование AI Agent + AI Agent Tool / Call n8n Workflow Tool
Router Категории формализуемы: support / billing / docs LLM classify + Switch
Pipeline Стадии: research → write → review Цепочка sub-workflows
Reflexion Нужен цикл правок Worker + Critic + IF + max iterations

Switch (deterministic routing) дешевле agent-routing, когда категория известна (Production AI Playbook, n8n.io). Multi-agent по умолчанию sequential; параллель - Execute sub-workflow или HTTP Request.

Схема workflow:
Chat Trigger → Orchestrator AI Agent → AI Agent Tool (Specialist A | B | C)
↘ Call n8n Workflow Tool → Sub-workflow workers → IF quality → Human approval → ответ

Делайте: для ticket type - Router + Switch. Не делайте: LLM-routing, если CRM или keyword уже даёт категорию.

Соберите orchestrator и sub-agents пошагово

  1. Шаг 1: Prerequisite. Single agent из B02 отвечает в Chat UI. Иначе сначала B02.
  2. Шаг 2: Orchestrator. Chat Trigger → AI Agent → Chat Model. В System Message перечислите sub-agents по зонам ответственности.
  3. Шаг 3: Specialists (MVP). 2-3 AI Agent Tool sub-nodes: своя модель, prompt, 1-3 tools каждый.
  4. Шаг 4: Tool description. 2-4 предложения: что делает, вход, выход, когда не вызывать. Без description orchestrator не делегирует.
  5. Шаг 5: JSON task envelope. Структурированный контекст между agents:
{
  "task": "найти тариф в базе знаний",
  "context": {"client_id": "12345"},
  "expected_output": "ответ с ссылкой на регламент"
}
  1. Шаг 6: Prod. Call n8n Workflow Tool вместо embedded AI Agent Tool - отдельный workflow на worker.
  2. Шаг 7: Лимит tools. Не больше 5 на agent; при 15 tools и использовании ~3 - split на focused agents (n8n playbook).
  3. Шаг 8: Continue on Error на внешних API tools.
Критерий AI Agent Tool Call n8n Workflow Tool
Canvas Sub-node на одном экране Отдельный workflow
Когда MVP Prod, версии, команда
Тесты Общий log Изолированный тест worker

Sub-agent vs MCP tool: у sub-agent свой prompt, LLM и memory; MCP (стандарт подключения сервисов к модели) - только tools без отдельной роли.

Делайте: два specialists (RAG-support + CRM). Не делайте: embedded agents на prod без плана sub-workflows.

Настройте память и session ID между агентами

Simple Memory или Postgres на orchestrator хранит диалог. У specialists:

  • Shared session ID: видит историю orchestrator - continuity, но лишние tokens.
  • Isolated session ID: только JSON из tool call - безопаснее и дешевле.

Между agents передавайте file_id или ссылку на документ, не full content PDF - иначе tokens съедают бюджет быстрее single-agent baseline. API (интерфейс связи n8n с CRM, почтой, LLM) credentials храните только в Credentials, не в открытом тексте workflow.

Делайте: isolated для CRM/billing. Не делайте: shared memory без явной нужды.

Подготовьте multi-agent систему к production

Reflexion: worker + critic, IF на quality, max iterations у critic. Human-in-the-loop перед оплатой, удалением, массовой рассылкой. ~40% пилотов автономных агентов отменяются из-за скрытых затрат - мерьте tokens с первого дня.

Cloud Starter n8n - от 20 €/мес при годовой оплате, 2 500 executions; один execution = полный прогон workflow, не каждый step (n8n.io/pricing, июнь 2026). Trial Starter/Pro без карты. Business self-hosted - Queue mode (несколько worker instances) при нагрузке. Self-host (свой сервер) даёт контроль данных в EU (Frankfurt на Cloud) - актуально для регламентов компании.

Простой CRM-routing без RAG и sub-workflows часто быстрее закрыть Make AI Agents из облака без Docker. Если multi-agent оказался тяжелее ожиданий - сравните оба стека на курсе Make.com и вайбкодинга (первые занятия бесплатны).

Делайте: error workflow на каждый sub-workflow. Не делайте: prod без max iterations у critic и эскалации к человеку.

Частые вопросы

Как работает AI Agent Tool в n8n?

Sub-node orchestrator: полный agent вызывается как tool через function calling. Добавьте AI Agent Tool, пропишите description, протестируйте делегирование в Chat UI.

Чем multi-agent отличается от одного агента с десятью tools?

Разные prompt, модель, память - меньше tool noise. Цена - coordination и ~15x tokens vs single-agent (Anthropic/n8n). Split agents при tool noise, не "для красоты".

Сколько sub-agents можно связать?

MVP: 2-5. При 10 agents - 45 связей overhead. Третий specialist - только с новой зоной ответственности.

AI Agent Tool или sub-workflow для production?

Tool - MVP на canvas. Call n8n Workflow Tool - prod: изолированные тесты, версии, error workflow.

Как разделить память между агентами?

Isolated session + JSON в tool call для CRM/billing. Shared - когда specialist должен видеть всю историю. Зафиксируйте выбор письменно.

Когда Switch вместо orchestrator?

Категория формализуема: ticket type, keyword, поле CRM. LLM classify → Switch → workflow. Agent-routing - для неоднозначных формулировок.

Типичные ошибки multi-agent в n8n?

Нет descriptions, лишний shared memory, critic без max iterations, embedded agents на prod. Пройдите чеклист из 16 пунктов и 30 test-кейсов до клиентов.
