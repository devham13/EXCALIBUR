Agent в Cursor отвечает по-разному в каждом проекте, потому что вы не зафиксировали правила: язык, стиль кода, запрет на выдуманные API. Cursor Rules – это короткие инструкции в файлах .mdc внутри папки .cursor/rules. За 20–40 минут вы создадите general.mdc и правило под свой стек, проверите срабатывание в Agent и перенесёте устаревший .cursorrules без конфликтов.

TL;DR / Быстрый инсайт: Project rules хранятся в .cursor/rules/*.mdc с YAML frontmatter (alwaysApply, globs, description). Создайте general.mdc с alwaysApply: true и stack-правило с globs под ваши файлы. Проверяйте в новом чате Agent, не в Tab. Rules задают поведение, а MCP в Cursor даёт доступ к внешним сервисам. Битый frontmatter или параллельный .cursorrules – частая причина "молчащих" правил.

Cursor Rules – подсказки для Agent (чата с ИИ), а не жёсткий компилятор. Нужны файлы .mdc с YAML между ---; plain .md в .cursor/rules игнорируется. Rules задают "как писать", MCP – "откуда брать данные".

1. Проверьте legacy и выберите формат .mdc

Файл .cursorrules в корне – legacy; новый стандарт – модульные .mdc с frontmatter. Если рядом оба формата, по опыту сообщества побеждает .mdc без предупреждения. Делайте: проверьте корень и Settings → Rules. Не делайте: править legacy и .mdc параллельно.

2. Создайте папку .cursor/rules и разберите frontmatter

Project rules хранятся в Git. Создайте каталог в корне:

mkdir -p .cursor/rules

Ключи frontmatter: alwaysApply: true – в каждом чате Agent (globs игнорируются); globs – паттерны через запятую; description – для intelligent-режима; без globs/description – только @rule-name. Держите правило короче 500 строк. В 401 публичном репозитории (UC Irvine, март 2026) 28,7% строк rules – дубликаты. Делайте: короткие файлы. Не делайте: alwaysApply на длинных текстах.

3. Настройте первые правила пошагово

Ниже – единый сценарий от пустого репозитория до рабочего набора. Выполняйте шаги по порядку.

  1. Шаг 1: Проверьте наличие .cursorrules в корне. Если есть – запланируйте миграцию в general.mdc, не редактируйте legacy параллельно с .mdc.
  2. Шаг 2: Создайте .cursor/rules и добавьте папку в Git (git add .cursor/rules).
  3. Шаг 3: Откройте Command Palette (Ctrl+Shift+P на Windows, Cmd+Shift+P на Mac) → команда "New Cursor Rule" или Settings → Rules, Commands → + Add Rule. Имя файла: general.mdc.
  4. Шаг 4: В frontmatter поставьте alwaysApply: true. В теле – 5–15 пунктов: язык ответов, минимальный diff, запрет выдумывать версии API.
  5. Шаг 5: Создайте второй файл, например react-components.mdc или content-markdown.mdc, с alwaysApply: false и globs под ваш стек (**/*.py, **/*.tsx, **/*.md).
  6. Шаг 6: Опционально intelligent-rule: description без globs для редких workflow.
  7. Шаг 7: Новый чат Agent → задача с glob-файлом → сравните стиль.
  8. Шаг 8: Не сработало – @rule-name в чате.
  9. Шаг 9: git commit .mdc и строка в README для команды.

Пример general.mdc:

---
alwaysApply: true
---

- Язык ответов: русский
- Не выдумывать API и версии – сверяться с официальной docs
- Минимальный diff; не трогать unrelated файлы

Пример stack-правила для контента:

---
description: Blog and markdown content rules
globs: **/*.md, content/**/*
alwaysApply: false
---

- Lead: боль + ответ + результат для читателя
- Цифры только с указанием источника

Делайте: копировать шаблоны и менять globs под свои папки. Не делайте: ставить alwaysApply: true на stack-правила – так вы перегрузите каждый чат лишними инструкциями.

4. Сравните четыре режима активации

Cursor (docs, 20.06.2026) выделяет четыре способа, когда правило попадает в контекст Agent. Выбор режима важнее объёма текста в body.

Режим в UI Frontmatter Когда срабатывает Когда выбирать
Always Apply alwaysApply: true Каждый чат Agent Язык, безопасность, общий стиль (коротко)
Apply to Specific Files alwaysApply: false + globs Когда Agent читает matching file React, Python, markdown, тесты
Apply Intelligently alwaysApply: false + description Agent оценивает задачу Редкие workflow без жёстких globs
Apply Manually без globs и description Только @rule-name в чате Эксперименты, черновики правил

Glob-rules подключаются, когда Agent читает matching file, не с первого сообщения. Rules работают только в Agent (Chat), не в Tab и Inline Edit. Делайте: тест на реальном файле. Не делайте: ждать rules в Cmd/Ctrl+K.

5. Свяжите Rules, MCP и Agent в один workflow

Rules – стиль и ограничения в репозитории. MCP – внешние tools (браузер, GitHub, CMS). Без rules Agent вызовет tool, но ответит хаотично.

Workflow настройки:
Выбор типа правила → файл .mdc → git commit → подключение MCP по гайду MCP Cursor → новый чат Agent → задача с glob-файлом → при необходимости @rule → проверка вызова MCP tool

Приоритет: Team Rules → Project Rules → User Rules. CLAUDE.md всегда в контексте. Делайте: 2–3 project rules, потом MCP. Не делайте: копировать чужие rules с GitHub без проверки YAML.

6. Пройдите миграцию и почините типичные сбои

Миграция: текст .cursorrules → general.mdc с alwaysApply: true → удалить legacy → commit. Альтернатива: /create-rule в Agent.

Чек-лист миграции и проверки (10+ пунктов):

  • Расширение файла именно .mdc, не .md
  • Открывающий и закрывающий --- в frontmatter на месте
  • YAML без табов; отступы пробелами
  • alwaysApply: true только у короткого general
  • Globs совпадают с реальными путями в репозитории
  • Несколько паттернов в globs разделены запятой
  • Legacy .cursorrules удалён или не правится параллельно
  • Тест в новом чате Agent, не в старой ветке диалога
  • Для glob-правила Agent реально открыл matching file
  • При сомнении вызван @rule-name вручную
  • User Rules не ожидаются в Inline Edit – это норма
  • Изменения .mdc подхватываются без перезапуска; при странном поведении – новый чат

Symptom → fix: молчит → closing --- и конфликт .mdc/.cursorrules; glob мимо → паттерн + Agent читает файл; лишний контекст → убрать alwaysApply с длинных правил.

Делайте: тестовый PR после миграции. Не делайте: ждать 100% соблюдения – rules это подсказки.

7. Зафиксируйте результат и двигайтесь дальше

  1. general.mdc + одно stack-правило под основной тип файлов.
  2. MCP подключите после rules, если Agent нужен доступ к внешним сервисам.
  3. README для команды: какие rules обязательны.
  4. Раз в квартал пересматривайте alwaysApply – лишняя строка = налог на контекст.

Связка Cursor + Make – курс по Make.com: rules в IDE, Make – фоновая публикация.

Частые вопросы

Как написать cursor rules mdc?

Создайте файл .cursor/rules/имя.mdc. Между двумя строками --- укажите YAML: alwaysApply, globs или description. Ниже --- напишите инструкции списком в Markdown. Сохраните с расширением .mdc и проверьте в новом чате Agent.

Чем cursor rules отличается от .cursorrules?

.cursorrules – один legacy-файл в корне без scoping. .mdc – модульные правила с frontmatter: можно включать инструкции только для нужных файлов и режимов. Cursor рекомендует мигрировать в .mdc с alwaysApply для бывшего содержимого .cursorrules.

Какие globs использовать для правил cursor?

Для React: src/**/*.tsx или **/*.tsx. Для Python: **/*.py или src/**/*.py. Для контента: **/*.md и content/**/*. Несколько паттернов – через запятую в одной строке globs. Сверьте паттерн с реальным путём файла, который Agent откроет в задаче.

Почему правило cursor не применяется?

Проверьте: расширение .mdc, закрывающий --- во frontmatter, нет параллельного .cursorrules, вы тестируете в Agent а не Tab, для globs Agent прочитал matching file. При битом YAML правило не загружается без явной ошибки – типичный silent failure по опыту форума Cursor.

Что делает alwaysApply: true?

Правило попадает в каждый чат Agent. Поля globs и description при этом игнорируются. Используйте только для коротких глобальных стандартов; длинные инструкции переносите в glob- или description-правила с alwaysApply: false.

Нужен ли перезапуск Cursor после правки .mdc?

Обычно изменения подхватываются без рестарта. Если поведение не меняется – откройте новый чат Agent или упомяните правило через @rule-name. Полный перезапуск нужен редко, в основном при смене Team Rules с dashboard.

Rules работают с Inline Edit (Cmd+K)?

Project Rules действуют в Agent (Chat). User Rules не применяются к Inline Edit. Tab autocomplete rules тоже не использует. Для проверки настройки всегда используйте чат Agent с задачей на ваш код или markdown.
