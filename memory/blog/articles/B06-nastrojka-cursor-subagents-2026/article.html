Agent в Cursor по умолчанию делает задачи по очереди: пока ищет файлы, не запускает тесты. Субагенты (cursor subagents) – узкие помощники с отдельной памятью, которые работают параллельно. За 20–30 минут вы создадите двоих в `.cursor/agents/`: reviewer только смотрит код, test-runner гоняет тесты. Настроите вызов `/name`, параллель из Agent Mode и правила, чтобы git не сломался.

TL;DR / Быстрый инсайт: Субагенты Cursor – независимые ИИ-помощники с собственным контекстом (с версии 2.4, январь 2026). Кастомный subagent – markdown с YAML-шапкой в `.cursor/agents/`. Встроенные explore, bash и browser работают без настройки. Начните с 2–3 ролей, readonly для аудита, явный `/name` или автоделегирование через description. Пять subagents параллельно ≈ пять диалогов по токенам – разведите file ownership до commit.

Субагент – узкий специалист в отдельном "кабинете": своя память, чистый главный чат. Главный Agent – оркестратор: он решает, кому отдать поиск, shell или review. Поле description в YAML – "визитка" для автоделегирования: чем конкретнее формулировка, тем чаще Agent вызовет нужную роль без `/name`. Если subagentу нужен браузер или CMS, сначала MCP по гайду MCP в Cursor – subagents наследуют MCP-tools родителя (кроме cloud subagents).

1. Сравните субагентов и Agent Mode: когда параллелить

Один Agent хватит для линейных задач: "найди → исправь → проверь". Subagents нужны, когда ветки независимы: review diff, тесты и поиск по репо одновременно. С Cursor 2.4 Agent делегирует встроенным explore (поиск), bash (терминал) и browser (MCP).

Workflow:
Задача → Agent → Task tool → subagent(s) параллельно → отчёты → вы принимаете diff

Делайте: параллелить независимые куски (разные файлы или readonly + правки). Не делайте: десятки generic subagents – Agent не поймёт, когда делегировать (docs: 2–3 focused роли).

2. Проверьте встроенные explore, bash и browser

До своих файлов дайте промпт: "Найди API key в проекте и покажи npm-скрипты в package.json". Должны сработать explore и bash без `.cursor/agents/`. Explore по умолчанию на более быстрой модели – удобно для параллельных поисков. Browser требует MCP browser tool и зелёный статус сервера в Tools & MCP.

Делайте: один тест explore + bash и зафиксируйте, что Agent Mode включён (не Ask, не Plan). Не делайте: дублировать explore кастомным search-agent – лишние токены и путаница в автоделегировании.

3. Настройте `.cursor/agents/` и YAML frontmatter

Файл subagent – YAML-шапка + инструкции. Проект: `.cursor/agents/`; глобально: `~/.cursor/agents/`. Проектный файл побеждает при конфликте имён.

Поле По умолчанию Когда менять
name имя файла Вызов `/name`
description – Автоделегирование; "use proactively" для test-runner
model inherit Другая модель, если разрешено планом
readonly false true – без прав на файлы и shell
is_background false true – прогресс в `~/.cursor/subagents/`

Делайте: коммитьте `.cursor/agents/` в git. Не делайте: JSON из сторонних гайдов – канон только `.md` + YAML (docs).

4. Создайте code-reviewer и test-runner пошагово

Cursor 2.4+, Agent Mode. Шаги:

  1. Шаг 1: Папка `.cursor/agents/` в корне проекта.
  2. Шаг 2: `code-reviewer.md` с `readonly: true`.
  3. Шаг 3: `test-runner.md` с "use proactively" в description.
  4. Шаг 4: Сохраните – перезапуск не нужен.
  5. Шаг 5: Тест: `/code-reviewer проверь последние изменения` – без записи в файлы.
  6. Шаг 6: Параллель: "Review API changes and update tests in parallel" – два Task-потока.
  7. Шаг 7: Commit `.cursor/agents/`; разведите зоны файлов.

Файл code-reviewer:

---
name: code-reviewer
description: Use when reviewing diffs, PRs, or recent code changes. Readonly audit for security and style.
model: inherit
readonly: true
is_background: false
---

You are a code reviewer. Check recent changes for secrets, injection, style, missing tests.
Do NOT edit files. Report as a numbered list with file paths.

Файл test-runner:

---
name: test-runner
description: Use proactively after any code changes. Run tests, fix failures, re-run until green.
model: inherit
readonly: false
is_background: false
---

When invoked: detect test command, run tests, fix failures minimally, re-run until green.
Summarize command, pass/fail, files changed.

Дальше добавьте verifier (readonly) – проверка "сделано vs заявлено". Делайте: инструкции в теле на английском. Не делайте: `readonly: false` на reviewer.

5. Вызовите `/name` и параллельный запуск

Явно: `/code-reviewer проверь auth flow` – когда нужна конкретная роль без риска ошибочного выбора. Авто: Agent читает description и вызывает Task tool; фразы "use proactively" и "always use for" усиливают триггер (официальная docs). Foreground subagent блокирует родителя до отчёта – удобно перед merge. Background (`is_background: true`) возвращает управление сразу; прогресс в `~/.cursor/subagents/`, можно resume по agent ID. С Cursor 2.5+ subagents запускают дочерних (лимит вложенности; у subagent второго уровня дальше нельзя).

Делайте: две независимые цели в одном сообщении – параллельные Task calls. Не делайте: параллельные правки одного файла – merge conflict.

6. Оркестрируйте: file ownership и worktrees

Пять subagents ≈ пять context windows и ~5× tokens (docs) – закладывайте бюджет, если гоняете review, tests и explore одновременно. Review – readonly на весь diff; test-runner – тестовые файлы и код, который чинит падения; две фичи – разные каталоги или отдельные git worktrees, чтобы агенты не трогали одни paths. Остановка родителя останавливает дочерних subagents. Cloud Agents: `/in-cloud` отправляет задачу на VM/branch, `/babysit` ведёт итерации по PR; MCP там из team config на cursor.com/agents, не из вашей локальной mcp.json.

Делайте: background для long tasks. Не делайте: shell-оркестратор как единственный путь – для старта хватит Agent + `.cursor/agents/`.

Критерий Subagents Background Agents MCP-tools
Суть Агент с промптом и моделью Долгая cloud/automation задача Доступ к API, браузеру, GitHub
Изоляция Свой context window Отдельная сессия Tool внутри одного агента
Параллель Несколько Task в одном сообщении Фон без блокировки Последовательные вызовы
Когда Review + tests + search PR на cloud, babysit Внешние данные

Вердикт: Subagents – параллельные роли в IDE. Background – работа на часы в cloud. MCP – внешние сервисы; subagents наследуют tools. Skills дополняют, но не заменяют изоляцию контекста.

8. Исправьте типичные ошибки

Subagent не в Task tool – проверьте путь `.cursor/agents/name.md`, YAML между `---`, режим Agent (не Ask). Vague description "helps with coding" – anti-pattern: Agent не делегирует.

  1. Шаг 1: Нет в Task tool – путь, YAML, Agent mode, конкретный description.
  2. Шаг 2: name в frontmatter = файл = `/name`.
  3. Шаг 3: Wrong model – оставьте inherit или смените в настройках.
  4. Шаг 4: Review правит файлы – `readonly: true`.
  5. Шаг 5: Дорого – меньше параллели; помните про 5× tokens.

Делайте: hooks subagentStart/subagentStop для логов. Не делайте: JSON-формат из EN-блогов.

Частые вопросы

Как создать субагента в Cursor?

Markdown в `.cursor/agents/имя.md`: YAML (name, description, model, readonly, is_background) + инструкции. Сохраните и вызовите `/имя задача` в Agent Mode.

Чем subagents отличаются от Cursor Rules?

Rules – статический контекст главного агента. Subagents – отдельное окно и Task tool. Rules – стиль; subagents – параллельные роли.

Почему субагент не появляется в Task tool?

Проверьте путь, YAML, Agent mode, description без vague "helps with coding". Смотрите hooks/policy на subagentStart.

Нужен ли Cursor 2.4+?

Да, релиз 22.01.2026. Help → About. Nested subagents – с 2.5+.

Foreground или background?

Foreground – отчёт до следующего шага. Background – long tasks, прогресс в `~/.cursor/subagents/`.

Subagents или Skills?

Skills – повторяемые процедуры в `.cursor/skills/`. Subagents – изоляция и параллель. Skills для чеклистов, subagents для review + tests одновременно.

Можно ли параллелить правки одного файла?

Нет – merge conflict. Один subagent на файл, readonly audit на diff, или git worktrees.
