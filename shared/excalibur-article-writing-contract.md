# Excalibur BLOG — контракт написания SEO/GEO статей

**Utility-only:** публикуем только практические материалы (how-to, чеклист, comparison, workflow). См. `shared/editorial-utility-only.md` и `memory/brief/editorial-policy.json`. Режим **A запрещён**.

Excalibur BLOG следует этому контракту для каждой статьи из `memory/topics/blog-topics.md`.

## Вход (обязательно прочитать)

- `memory/topics/blog-topics.md` — карточка темы (`topic_id`, H1, queries, H2/H3, FAQ, links)
- `memory/brief/site-brief.md` — бренд, сайт, тон, запреты
- `memory/brief/fact-bank.md` — проверенные факты
- `memory/brief/conversion-map.md` — CTA, лимиты упоминаний
- `shared/excalibur-article-writing-contract.md` (этот файл)

Перед текстом — research → `memory/blog/articles/<topic_id>-<slug>/research-notes.md`.

## Голос автора

Не используй фиксированную персону. Голос бери из:

1. `memory/brief/site-brief.md` — если задан `author_voice` или эксперт;
2. иначе — **редакция бренда** из site-brief: практично, по-человечески, без корпоративной воды;
3. стиль: как объясняешь знакомому коллеге — живо, по делу, умеренная ирония допустима.

От первого лица («я») — **только** если в brief явно editorial first-person или режим B (личный опыт). Иначе — «мы в [бренде]» / нейтрально-экспертно без фейкового «я восстановил код за выходные».

## Параметры

- **Язык:** русский
- **Объём:** **9 000 символов с пробелами** (допустимо **8 500–9 500**)
- **Оптимизация:** SEO + GEO (Google AI Overviews, Яндекс Нейро, answer blocks) + сильный первый экран (RSS/сниппет)
- **Актуальность:** цены, даты, версии продуктов — только из research-notes / fact-bank / явного web research на дату прогона. Нет цифры — не выдумывай.

## Ключевые слова

Из карточки темы в `blog-topics.md`: `primary query`, `secondary queries`.

Правила:

- Вплетай только релевантные запросы к H1 и теме.
- Запрещено: чужие продукты/игры ради SEO, пиратство/взлом/торренты, если статья не про это.
- Каждое вхождение — осмысленное; лучше пропустить, чем набить.

## Фактура (Data SEO)

Приоритет: `research-notes.md` → `fact-bank.md` → осторожные формулировки («в открытых источниках…», «на момент публикации…»).

Запрещено выдумывать проценты «по исследованиям», бюджеты, сроки, точные цены API без источника.

## Режим статьи (utility-only)

**Только B) Инструкция / гайд / чеклист / comparison** — задача → шаги → ошибки → инструменты → вывод.

**A) Новости / кейс / разбор — не публикуем** в этом блоге.

Каждый H2 обязан нести **действие**: что сделать / не делать и почему. Абзац без рекомендации — удалить.

## Стиль и доступность (Человеческий язык для всех)

**КРИТИЧЕСКОЕ ПРАВИЛО:** Мы пишем простым, понятным и живым языком. Нашу статью должен легко прочитать и до конца понять абсолютно любой человек (независимо от того, программист он, предприниматель, гуманитарий или новичок). 

1. **Без профессионального снобизма:** Избегай сложных сухих конструкций и заумных терминов, если их можно выразить простыми словами.
2. **Обязательная расшифровка:** Если в силу специфики темы без термина обойтись невозможно (например: *RAG*, *API*, *Self-hosted*, *Векторная база данных*), ты **обязан сразу же расшифровать и объяснить его суть** простыми словами на понятном примере «на пальцах».
3. **Без воды:** Каждое предложение — факт, цифра, мнение или действие.
4. **Ритм:** Чередуй короткие и длинные фразы; абзацы до 3–6 предложений.
5. **Натуральность:** «тут важный момент», «честно говоря» — умеренно.
6. **Не более 2-3 легких шероховатостей** на статью (самоправка). Без орфографических ошибок.
7. **В каждой секции** — минимум одна рекомендация: что делать / не делать и почему.

## Запрещено

- Эмодзи в тексте
- Длинные тире («—») и кавычки-ёлочки (««», «»»). Всегда используйте среднее тире («–») или короткий дефис («-»), а вместо кавычек-ёлочек — прямые двойные кавычки («"»).
- Слово **VPN** и «обход блокировок», «зеркала», «прокси для обхода» — не упоминать. Про доступ к сервисам: нейтрально «зависит от региона и аккаунта; смотрите официальные доки».
- Штампы: «в современном мире», «давайте разберёмся», «подводя итог», «в заключение», «стоит отметить», «не секрет что», «на сегодняшний день»
- «Гарантия успеха», «лучший курс», «написано ИИ», мета про «стратегию текста»
- Placeholders, lorem, «пример отзыва», фейковые отзывы

## Структура HTML

### Блок 1: Хук + Direct Answer (первый `<p>`)

350–500 символов. Формула: «[боль/интрига] — [суть ответа] — [что получит читатель]». Без «в этой статье вы узнаете».

### Блок 2: Инсайт-выжимка (AEO TL;DR Box)

Обязательный блок `<blockquote>` сразу после первого `<p>` (Lead). Он содержит концентрированные выводы по статье в 3-4 предложениях для голосового и нейропоиска. 
Пример: `<blockquote><b>TL;DR / Быстрый инсайт:</b> Суть проблемы простыми словами...</blockquote>`

### Блок 3: Оглавление (Table of Contents) — **ЗАПРЕЩЕНО**

**Не вставляй** в `article.html` кликабельное оглавление: нумерованный или маркированный список с якорными ссылками на заголовки (`<ol><li><a href="#section-id">...</a></li></ol>`).

**Почему:** при RSS-синдикации в Яндекс.Дзен такой блок дублируется в начале материала и выглядит как лишняя «простыня» ссылок перед текстом. На WP-теме навигация по статье не нужна.

**Структура после TL;DR:** сразу идут 1–2 абзаца контекста (`<p>`), затем первый `<h2>`.

**Запрещённый паттерн (не использовать):**
```html
<!-- ЗАПРЕЩЕНО -->
<ol>
  <li><a href="#what-is-mcp">Разберите, зачем вам MCP</a></li>
  <li><a href="#config-location">Выберите, где хранить mcp.json</a></li>
</ol>
```

Атрибут `id` у `<h2>` допускается для внутренних ссылок из других статей, но **не** для оглавления в теле материала.

### Блок 4: Контекст (1–2 `<p>`)

Почему тема важна сейчас; для режима A — без «я сам это сделал».

### Блок 5: Основной контент (5–8 секций, `<h2>`/`<h3>`)

~70% объёма. Раскрой обещание H1 из `blog-topics.md`. Каждому `<h2>` при необходимости можно задать латинский `id` для внешней перелинковки (не для оглавления в теле): `<h2 id="why-n8n">1. Почему n8n...</h2>`.

На секцию: суть → факты → типичная ошибка → рекомендация.

**Разрешённые теги:** `<h2>`, `<h3>`, `<p>`, `<b>`, `<i>`, `<a>`, `<ul><li>`, `<ol><li>`, `<blockquote>`, `<table>`, `thead`, `tbody`, `tr`, `th`, `td` (только для структурированных сравнений и ТХ).  
**Запрещено:** `<h1>`, `<h4>`, `<div>`, `<span>`, `<strong>`, `<em>`, `<u>`, `<s>`, `<details>`.  
Внутри `<li>` — только текст.

### Блок 6: Верификация данных и E-E-A-T (Fact Check Box)

В самом конце статьи (перед FAQ) вставляется семантический блок `<blockquote>`, подтверждающий экспертность и достоверность фактов.
Формула:
```html
<blockquote>
  <b>Материал проверен:</b> эксперт Елена Ковалева (Главный эксперт по SEO/GEO).<br>
  <b>Достоверность данных:</b> все статистические показатели и ключевые фразы верифицированы по данным Яндекс Вордстат на июнь 2026 года.
</blockquote>
```

## Правила динамического контента (Tables, Images, Diagrams, Comparisons) и RSS-безопасность (Яндекс.Дзен)

**КРИТИЧЕСКИЙ СТАНДАРТ СОВМЕСТИМОСТИ (RSS / Яндекс.Дзен):** 
Поскольку статьи автоматически передаются в Дзен по RSS-ленте, тело статьи `article.html` должно состоять строго из чистого, семантического HTML. Любые нестандартные кастомные теги, скрипты (JS), интерактивные кнопки или инлайновые стили `style="..."` будут вырезаны Дзеном и сломают отображение. 

1. **Код и JSON-сценарии:** Оформляйте строго через стандартную связку `<pre><code>код_здесь</code></pre>`. Запрещено вшивать кастомные JS-кнопки «Копировать код» внутрь статьи. WordPress-тема на сайте автоматически добавит кнопку копирования поверх `<pre><code>` с помощью глобального скрипта, а Дзен получит чистый валидный код, который читатель сможет скопировать вручную.
2. **Выжимка TL;DR (Инсайт в начале статьи):** Оформляйте строго через стандартный тег цитаты `<blockquote>`. Дзен нативно и очень красиво стилизует цитаты на своей платформе, а на сайте WP мы настроим рамку через CSS темы.
3. **Оглавление в теле статьи:** **Запрещено.** Не добавляйте списки с якорными ссылками на H2 (`<ol><li><a href="#...">`). После TL;DR сразу идёт текст, без «простыни» навигации.
4. **E-E-A-T автора:** Шапка автора (аватар, био, дата) формируется на уровне темы WordPress и не должна жестко хардкодиться в HTML статьи. В Дзен она улетит через стандартные RSS-заголовки.

### 1. Таблицы (`<table>`) — Обязательны для Comparison и ТХ

- **Когда:** Любая тема с интентом `comparison` (сравнение), обзоры инструментов, сравнение тарифов или детальные технические характеристики.
- **Стандарт:** Минимум **1 сравнительная таблица** на статью.
- **Структура:** Таблица должна иметь `<thead>` с четкими заголовками (например: *Параметр*, *Наш сервис*, *Альтернатива А*, *Альтернатива Б*) и не менее 3 строк `<tr>` в `<tbody>`. Данные должны быть конкретными (цифры, бинарные "Да/Нет", четкие преимущества).
- **Пример:**

```html
<table>
  <thead>
    <tr>
      <th>Критерий сравнения</th>
      <th>Вайбкодинг (Cursor/Claude)</th>
      <th>Классический Scratch</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><b>Скорость сборки игры</b></td>
      <td>15–30 минут (через ИИ-промпты)</td>
      <td>2–4 часа (ручное перетаскивание блоков)</td>
    </tr>
    <tr>
      <td><b>Сложность синтаксиса</b></td>
      <td>Отсутствует (общение на русском языке)</td>
      <td>Низкая (визуальные блоки-пазлы)</td>
    </tr>
  </tbody>
</table>
```

### 2. Сравнения и вердикты (`<blockquote>` + `<b>`)

- **Когда:** В обзорах и сравнениях.
- **Стандарт:** Каждое сравнение должно завершаться понятным, выделенным выводом (вердиктом).
- **Формат:** Используй тег `<blockquote>` для оформления рамки экспертного вердикта. Внутри вердикта жирным шрифтом выделяй рекомендацию под конкретную аудиторию.
- **Пример:**

```html
<blockquote>
  <b>Итоговый вердикт:</b> Если вашему ребенку 7-9 лет и он никогда не сталкивался с логикой программ — начните со Scratch. Для детей от 10 лет <b>вайбкодинг</b> даст в 10 раз больше вовлечения, так как позволит сразу создавать взрослые, реальные игры.
</blockquote>
```

### 3. Изображения (`<img>`) и иллюстрации

- **Когда:** Пошаговые руководства (`how_to`), обзоры интерфейсов софта или демонстрация маскотов бренда.
- **Правило вставки:** В статье допускается размещение **1–3 контекстных изображений** (не считая обложки-featured).
- **Источники:**
  - Если в `AURA_ASSET_REGISTRY.json` есть готовые тематические иллюстрации или маскоты — бери их относительные локальные пути.
  - Если статья про софт/инструмент, а скриншотов нет — вставляй плейсхолдер с четким ТЗ для дизайнера или генератора: `<img src="assets/placeholder-ui.png" alt="Скриншот интерфейса Cursor при создании игры" />`.
- **Подпись под картинкой:** Каждое изображение должно сопровождаться подписью внутри параграфа с наклонным шрифтом `<i>`.
- **Пример:**

```html
<p align="center">
  <img src="assets/cursor-interface-prompt.png" alt="Интерфейс ИИ-редактора Cursor при написании промпта для игры" />
  <br>
  <i>Иллюстрация 1: Процесс создания логики движения персонажа через текстовый запрос на русском языке</i>
</p>
```

### 4. Инфографика, текстовые схемы и диаграммы

- **Когда:** Описание сложных процессов, этапов воронки, архитектуры ИИ-ассистентов или роадмапов обучения.
- **Правило разметки:** Если в статье объясняется сложный процесс из нескольких шагов, Excalibur должен оформить его в виде текстовой схемы (с использованием стрелок-символов `→`, `↓` или ASCII-графики внутри `<blockquote>`) либо в виде структурированного вложенного списка.
- **Пример текстовой схемы:**

```html
<blockquote>
  <b>Схема работы вайбкодера:</b>
  <br>
  Идея игры (ребенок) → Описание на русском языке (Промпт) → ИИ-генерация кода (Claude) → Запуск и тест в браузере (Результат)
</blockquote>
```

### 5. Пошаговые инструкции (архетип `how_to`)

- **Когда:** Обучающие статьи, пошаговые настройки или чек-листы.
- **Стандарт:** Должен использоваться тег `<ol>` с нумерованными шагами `<li>`. Каждый шаг должен начинаться с активного глагола действия (например, *Шаг 1: Скачайте...*, *Шаг 2: Настройте...*).

---

### Блок 4: Мягкий CTA бренда (внутри одной секции блока 3)

2–3 предложения **только если** уместно по теме. Бери URL и формулировку из `conversion-tracking-map.md` / `offers-map.md`.  
Не больше **3** упоминаний основного офера/CTA на статью (включая баннер и «что дальше»).

### Блок 5: Inline-баннер (опционально)

**Только если** в `conversion-tracking-map.md` есть `excalibur_inline_banner: yes` и `banner_image_url` + `banner_link_url`.

Один раз после 3–4-й секции. **До баннера** в статье не должно быть других `<img>` (обложка — featured image в WP, не в body).

```html
<a href="..."><img src="..." alt="..." /></a>
<p><a href="...">Текст ссылки</a></p>
```

Если баннер не задан — пропусти блок.

### Блок 6: «Что дальше»

3–5 шагов + одно предложение с основным CTA (из conversion map). Без давления.

### Блок 7: Дополнительные ссылки

Не больше **2–3** внутренних ссылок из `internal links` темы + релевантные landing pages.

### Блок 8: FAQ (5–7 вопросов) — **ровно один блок**

В `article.html` допускается **только одна** секция FAQ:

```html
<h2>Частые вопросы</h2>
<h3>Вопрос по теме статьи?</h3>
<p>Ответ 2–4 предложения.</p>
```

**Запрещено:**
- второй блок FAQ с другим заголовком (`Часто задаваемые вопросы по теме`, `FAQ по теме` и т.п.);
- универсальные вопросы про «контент-завод / AI-агенты / нужно ли программировать», если они **не** относятся к primary query статьи.

**Почему:** тема WordPress на **production.com** может добавлять **глобальный** FAQ после контента поста. Excalibur пишет тематический FAQ один раз; дублирование на live-странице = баг темы, не Writer.

Вопросы — только по теме и ключам из `research-notes.md` / `faq_hints`, без мусора.

- Файл: `memory/blog/articles/<topic_id>-<slug>/article.html`

### schema.jsonld (Спецификация)

Файл содержит структурированные данные в формате JSON-LD (в виде одиночного объекта или структуры `@graph`):

1. **Базовый тип (`BlogPosting`):**
  - `headline`: заголовок статьи (H1).
  - `description`: краткое описание (meta description).
  - `datePublished`: текущая дата.
  - `image`: URL или путь к сгенерированной обложке.
  - `author`: объект `Person` с подробным описанием, строго соответствующим реестру авторов `shared/authors-registry.json` (по совпадающему `author_id` из `article.meta.json`):
    - `name`: имя эксперта из реестра (например, Александр Смирнов).
    - `jobTitle`: должность эксперта из реестра.
    - `worksFor`: название бренда (например, Maya AI).
    - `sameAs`: массив ссылок на авторитетные профили автора (LinkedIn, GitHub, Habr) из реестра для подтверждения E-E-A-T.
2. **Разметка частых вопросов (`FAQPage`):**
  - Добавляется, если в статье есть FAQ. `mainEntity` содержит массив `Question` и `Answer`, текст которых должен строго соответствовать видимому контенту в HTML.
3. **Разметка для пошаговых инструкций (`HowTo`):**
  - Добавляется для **Режима B** (Инструкция / гайд).
  - Содержит `step` (массив шагов с `name`, `text` и `url`), `totalTime`, `tool` и `supply` (из `research-notes.md`).
4. **Разметка для сравнений и обзоров (`Review` или `Product`):**
  - Добавляется для **Comparison** тем.
  - Содержит `itemReviewed` (тип `Product` или `Thing`), `reviewRating` (оценка, например 4.8/5.0), `publisher` и `reviewBody` (краткий вердикт).

Все поля должны быть валидными по валидаторам Schema.org / Google Rich Results. Валидный JSON-LD, без разрывов строк внутри строк, экранирование кавычек.
